## Project Overview

DisasterScan is a machine learning project that automatically determines whether a tweet is describing a **real disaster event** (fire, flood, earthquake, etc.) or using disaster-related language in a non-disaster context (metaphors, sarcasm, everyday language).

The project covers the full ML lifecycle:
- Exploratory Data Analysis
- Text cleaning and feature engineering
- Training and comparing multiple ML models
- Deep Learning with LSTM
- Final model evaluation and threshold tuning
- Production deployment via a Streamlit web app

---

## The Challenge

Social media platforms like Twitter are flooded with information during disasters, making real-time monitoring critical. However, many tweets use disaster-related words in non-disaster contexts:

| Tweet | Label |
|---|---|
| `"Wildfire spreads across 1000 acres in California"` | Real Disaster |
| `"That presentation was absolutely fire "` | Not a Disaster |
| `"Flood warning issued for coastal areas"` | Real Disaster |
| `"I'm flooded with work this week"` | Not a Disaster |

**Objectives:**
- Build a model that predicts whether a tweet is about a real disaster
- Minimise false alarms (high precision) and missed disasters (high recall)
- Handle slang, abbreviations, and cultural nuances
- Deploy a scalable real-time classification solution

## Models Used

### Classical ML (Notebook 2)

| Model | Description |
|---|---|
| `LinearSVC` | Linear Support Vector Classifier — fast, strong baseline for text |
| `LogisticRegression` | Probabilistic linear classifier |
| `MultinomialNB` | Naive Bayes for frequency-based text features |
| `RandomForestClassifier` | Ensemble of decision trees (100 estimators) |

**Text Representation:** TF-IDF
```
max_features = 5000
ngram_range  = (1, 2)   # unigrams + bigrams
min_df       = 5
max_df       = 0.8
```

### Deep Learning (Notebook 4)

```
Architecture:
  Embedding(vocab=10000, dim=128)
  LSTM(64 units)
  Dropout(0.5)
  Dense(32, relu)
  Dense(1, sigmoid)

Training:
  epochs        = 5
  batch_size    = 32
  val_split     = 0.2
  optimizer     = Adam
  loss          = binary_crossentropy
```

### Final Model Selection (Notebook 5)
The best performing model across all metrics is automatically selected and saved to `models/model_metadata.json`. The decision threshold is tuned in the range **0.30–0.80** to balance precision and recall, with a minimum clamp of **0.50** to prevent over-flagging.

## App Features

The DisasterScan Streamlit app has three tabs:

### Tab 1 — Single Tweet
- Type any tweet or choose from built-in examples
- Instant prediction: **[DISASTER]** or **[SAFE]**
- Confidence percentage and cleaned text preview
- **Plotly confidence gauge** showing prediction strength
- **Probability split bar** showing disaster vs safe probability

### Tab 2 — Batch CSV
- Upload a CSV file with a `text` column
- Classifies all tweets with a progress bar
- Summary metrics: total tweets, disaster count, safe count

### Tab 3 — History
- Colour-coded table of every prediction made in the current session
- Disaster predictions highlighted in red, safe in green

- Live model metrics: F1-score, Accuracy, Precision, Recall, ROC-AUC, Threshold

## Notebook Descriptions

### Notebook 1 — Data Exploration (EDA)
Explores the raw dataset to understand its structure, class distribution, and text characteristics.
- Loads `twitter_disaster.csv`
- Handles missing values
- Univariate, bivariate, and multivariate analysis
- Text length distributions, keyword frequencies
- Applies `clean_text()`: lowercase, remove URLs, mentions, hashtag symbols, punctuation, numbers
- Saves cleaned data to `data/train_prepared.csv`

### Notebook 2 — Feature Engineering & Model Training
Builds text features and trains four classical ML models.
- Creates TF-IDF feature matrix (5000 features, unigrams + bigrams)
- 80/20 stratified train-test split
- Trains LinearSVC, Logistic Regression, Multinomial NB, Random Forest
- Evaluates with accuracy, precision, recall, F1, cross-validation F1
- Selects best model by test F1-score
- Saves `tfidf.pkl` and `best_model.pkl`

### Notebook 3 — Model Insights & Feature Analysis
Deep-dives into the best ML model's behaviour.
- Confusion matrix and classification report
- ROC curve and AUC score
- Precision-Recall curve
- Top predictive words for disaster vs non-disaster
- Misclassified tweet analysis — word count distribution of errors
- Saves misclassified examples for review

### Notebook 4 — Deep Learning (LSTM)
Implements a sequence-based neural network for comparison with classical ML.
- Tokenizes text with Keras `Tokenizer` (vocab=10,000, maxlen=100)
- Architecture: Embedding → LSTM(64) → Dropout(0.5) → Dense(32) → Sigmoid
- Trains for 5 epochs with 20% validation split
- Evaluates on test set, computes F1-score
- Saves `lstm_model.h5` and `lstm_history.json`

### Notebook 5 — Final Model Evaluation & Comparison
Head-to-head comparison of all models on the same held-out test set.
- Unified metrics table: ML vs LSTM
- Side-by-side metrics bar chart
- LSTM accuracy curve (train vs validation)
- Decision threshold tuning (0.30–0.80 range, F1-optimised, min clamp 0.50)
- Final model selection
- Saves `model_metadata.json` and `lstm_tokenizer.pkl` for deployment

### Notebook 6 — Streamlit Deployment
Builds and launches the production web application.
- Loads all saved artefacts
- Validates prediction pipeline with smoke test
- Writes `deployment/app.py` (Streamlit UI)
- Auto-installs Streamlit and Plotly if missing
- Runs deployment checklist
- Launches app and opens `http://localhost:8501` automatically

## Key Findings

- **Disaster tweets** typically contain explicit disaster keywords (`fire`, `flood`, `earthquake`, `evacuation`) and tend to be longer with more URLs
- **Non-disaster tweets** frequently use disaster words metaphorically — this is the primary challenge
- **LinearSVC with TF-IDF** consistently achieves the best F1-score among classical ML models for text classification
- **LSTM** captures better sequential context but requires more data and tuning to outperform LinearSVC on this dataset size
- Misclassifications concentrate on **short, ambiguous tweets** (< 5 words) where context is minimal
- The optimal decision threshold is typically between **0.45–0.60**, not the default 0.5

## Known Limitations

| Limitation | Explanation |
|---|---|
| **Negation problem** | TF-IDF cannot understand "there is NO wildfire" — it sees "wildfire" and predicts Disaster |
| **Short tweet ambiguity** | Very short tweets (1–3 words) lack enough context for reliable classification |
| **Metaphorical language** | Phrases like "this is a disaster" in non-emergency contexts can confuse the model |
| **Threshold sensitivity** | The model is sensitive to the decision threshold — a low threshold over-flags as Disaster |
| **Dataset scope** | Trained only on English tweets; may not generalise to other languages or dialects |

These limitations are inherent to bag-of-words (TF-IDF) approaches. Transformer-based models (BERT, RoBERTa) would handle context and negation significantly better and represent a natural next step.
